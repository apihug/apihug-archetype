# RAD 业务架构设计

## 核心概念定义

### Customer（客户）
**定义：** 所有使用系统的**人类用户**

**包括的角色：**
- 管理员（Admin）
- 普通成员（Member）
- 员工（Employee）
- 经理（Manager）
- 以及其他所有业务角色

**特点：**
- 业务层面的"人"的概念
- 一个 Customer 可以有多个角色
- 一个 Customer 可以属于多个组织
- 一个 Customer 可以属于多个部门

**实体：** `CustomerEntity`
- 存储人的基本信息
- 包括：姓名、邮箱、手机号、状态等

---

### User（系统账户）
**定义：** 用于登录认证的**系统账户**

**特点：**
- 技术层面的"账户"概念
- 用于登录认证（用户名 + 密码）
- 一个 Customer 对应一个 User 账户
- 包含：用户名、密码哈希、状态等

**实体：** `UserEntity`（TODO: 未来扩展）
- 存储认证信息
- 与 Customer 是一对一关系

---

## API 设计原则

### Customer API（客户相关）
- `/api/customer/*` - 客户自身信息查询
- `/api/customers/*` - 客户管理（CRUD）

### User API（账户相关）
- `/api/auth/*` - 认证相关（登录、登出）
- `/api/users/*` - 系统账户管理

## 为什么这样设计？

### 避免混淆
- **Customer** = 业务的人（Customer、Member、Employee 都是 Customer 的角色）
- **User** = 技术的账户（用于登录认证）

### 扩展性
- 一个 Customer 可以有多个 User 账户（未来扩展）
- 支持第三方登录（一个 Customer 关联多个外部账户）
- 支持企业客户（一个 Customer 属于多个组织）
