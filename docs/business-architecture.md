# RAD 业务架构设计

## 核心概念定义

### Customer（客户 / 自然人）
**定义：** 所有使用系统的**人类用户**，登录主体

**特点：**
- 系统中唯一的“人”的概念
- 通过用户名 + 密码登录认证
- 一个 Customer 可以加入多个租户（Tenant）
- 在每个租户中拥有不同的角色和权限

**实体：** `CustomerEntity`
- 存储人的基本信息和认证信息
- 包括：用户名、密码哈希、邮箱、状态、默认租户 ID 等

---

### Tenant（租户 / 公司）
**定义：** 组织单元，资源隔离的核心边界

**特点：**
- 每个租户拥有独立的角色、菜单、部门、权限体系
- 平台（Platform）是特殊租户（`is_platform = true`），拥有全局管理权限
- 租户有最大成员数限制（`max_members`）

**实体：** `TenantEntity`
- 存储租户信息
- 包括：租户代码、租户名称、是否平台、描述、最大成员数、状态等

---

### TenantMember（租户成员）
**定义：** Customer 在 Tenant 中的身份关系

**特点：**
- 表示一个客户加入某个租户后的成员身份
- 拥有成员角色（MemberRoleEnum）：OWNER / ADMIN / MEMBER
- 可被分配到部门（`department_id`）
- 可被锁定（`is_locked`）
- 可设置为默认租户（`is_default`）

**实体：** `TenantMemberEntity`
- 关联 customer_id 和 tenant_id
- 包括：成员角色、部门 ID、是否锁定、是否默认等

---

## 业务关系

```
┌─────────────────────────────────────────────────────────┐
│                     Customer (客户/自然人)                    │
│  ┌─────────────────────────────────────────────────┐    │
│  │  登录主体：用户名 + 密码                              │    │
│  │  基本信息：姓名、邮箱、状态                            │    │
│  └─────────────────────────────────────────────────┘    │
│                        ↓ 1:N                             │
│  ┌─────────────────────────────────────────────────┐    │
│  │  TenantMember (租户成员)                               │    │
│  │  - 成员角色: OWNER / ADMIN / MEMBER                 │    │
│  │  - 部门分配: department_id                           │    │
│  │  - 锁定状态: is_locked                               │    │
│  └─────────────────────────────────────────────────┘    │
│                        ↓ N:1                             │
│  ┌─────────────────────────────────────────────────┐    │
│  │  Tenant (租户/公司)                                   │    │
│  │  - 资源隔离边界                                       │    │
│  │  - 拥有独立的角色、菜单、部门、权限                  │    │
│  │  - Platform = Tenant WHERE is_platform=true       │    │
│  └─────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘
```

## API 设计原则

### Customer API（客户相关）
- `/api/customer/*` - 客户自身信息查询（当前客户信息、租户列表、切换租户）
- `/api/customers/*` - 客户管理（CRUD）
- `/api/auth/*` - 认证相关（登录、登出）

### Tenant API（租户相关）
- `/api/tenants/*` - 租户管理（CRUD）
- `/api/tenants/{tenantId}/members/*` - 租户成员管理

### 资源管理 API（租户隔离）
- `/api/roles/*` - 角色管理（每个租户独立）
- `/api/menus/*` - 菜单管理（每个租户独立）
- `/api/departments/*` - 部门管理（每个租户独立）

## 为什么这样设计？

### 概念清晰
- **Customer** = 自然人（登录主体）
- **Tenant** = 公司/组织（资源隔离边界）
- **TenantMember** = 客户在租户中的身份（关联关系）
- **Platform** = 特殊租户（is_platform=true），而非独立概念

### 扩展性
- 一个 Customer 可以加入多个 Tenant
- 支持第三方登录（未来扩展）
- 租户间完全隔离，平台租户拥有全局管理能力
