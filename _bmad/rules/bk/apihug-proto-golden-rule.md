# ApiHug Proto Golden Rule

**Architectural Principle**: API = contract, Domain = persistence, connected via Service layer.

## MANDATORY EXECUTION RULES (READ FIRST):

- 🛑 **NEVER** add `page`, `size`, `sort` to Request Message — framework injects via `pageable: true`
- 🛑 **NEVER** wrap response in `PageableXxx` or `XxxList` — framework wraps `Result<Pageable<T>>`
- 🛑 **NEVER** use `string` for enum values — use `EnumType` with full package path
- 🛑 **NEVER** add file-level options (`java_package`, `go_package`) — build system injects
- 🛑 **NEVER** import `domain/` entities in `api/` layer — API layer must NOT depend on Domain
- 🛑 **NEVER** use SQL reserved words as column names — use prefixes: `user_name`, `order_no`
- 🛑 **NEVER** modify `_full.xml` migration file — auto-generated by wire
- 🛑 **NEVER** add redundant "ENUM" suffix — `STATUS_ACTIVE` ✅, `STATUS_ENUM_ACTIVE` ❌
- 🛑 **NEVER** use relative type references or import — always use full package reference and full path import, `com.example.Order order =1` ✅, `Order order = 1` ❌
- 🛑 **NEVER** create Request Message with only 1-2 fields — use `google.protobuf.Empty` + `parameters{}`
- 🛑 **NEVER** create proto message with single repeated field to support List, use `input_repeated` and `output_repeated` in method option.
- 🛑 **NEVER** use `DTO` when define request response message, ANY definition MUST hava real business meaning!
- 🛑 **NEVER** put all code in `api`, `domain`, `infra`, Each feature should create individual sub dir in those main dir!
- 🛑 **NEVER** define INNER message type
- ✅ **ALWAYS** run `gradlew :{module}:wire` after proto changes — sync generated code
- ✅ **ALWAYS** use `snake_case` for proto fields — convention compliance
- ✅ **ALWAYS** define enums in `infra/{module}_constant.proto` — centralized, shared
- ✅ **ALWAYS** use full package path: `com.example.user.infra.UserStatusEnum`
- ✅ **ALWAYS** add table prefix: `SYS_`, `BIZ_`, `CFG_`, `LOG_` — avoid reserved words
- ✅ **ALWAYS** use `pageable: true` for pagination — single source of config
- ✅ **ALWAYS** define errors with `code`, `http_status`, `phase`, `severity`

## 🚫 Prohibited Rules

### 1. No File-Level Options

**NEVER** add file-level options in `.proto` files. Build system injects them automatically.

```proto
// ❌ FORBIDDEN
option java_package = "...";
option java_multiple_files = true;
option go_package = "...";
```

### 2. No Cross-Layer References

**API layer (`api/`)** must NOT import **Domain layer (`domain/`)** entities.

```proto
// ❌ WRONG — API importing Domain
import "com/example/domain/entities/user.proto";
message CreateUserRequest {
  UserEntity user = 1;  // ❌ Domain entity in API layer
}

// ✅ CORRECT — API uses primitives + shared enums
message CreateUserRequest {
  string user_name = 1;
  UserStatusEnum status = 2;  // ✅ Enum from infra/
}
```


### 3. Pageable API — Zero Tolerance

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  PAGEABLE API DESIGN — ABSOLUTE RULES                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│  ✅ CORRECT                           │  ❌ FORBIDDEN                       │
├───────────────────────────────────────┼─────────────────────────────────────┤
│  pageable: true                       │  page/size/sort in Request Message  │
│  returns (UserSummary)                │  returns (PageableUserSummary)      │
│  message SearchRequest { keyword }    │  message SearchRequest { page }     │
└───────────────────────────────────────┴─────────────────────────────────────┘
```

- **Rule 1**: NEVER add `page`, `size`, `sort` to Request Message — framework injects them.
- **Rule 2**: NEVER wrap response in list/pageable type — framework wraps `Result<Pageable<T>>`.
- **Rule 3**: Use `pageable: true` as the ONLY way to enable pagination.

```proto
// ✅ CORRECT
rpc SearchUsers (SearchUsersRequest) returns (UserSummary) {
  option (hope.swagger.operation) = {
    post: "/users/search";
    pageable: true;  // ← ALL you need
  };
}

message SearchUsersRequest {
  string keyword = 1;  // Business filters only
  // NO page/size/sort
}

// ❌ FORBIDDEN
rpc SearchUsers (SearchUsersRequest) returns (PageableUserSummary);  // Double-wrapping
message SearchUsersRequest { int32 page = 1; int32 size = 2; }       // Manual pagination
```

### 4. Enum Must Be Independently Defined — ZERO TOLERANCE

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  ENUM DESIGN — ABSOLUTE RULES                                              │
├─────────────────────────────────────────────────────────────────────────────┤
│  ✅ CORRECT                           │  ❌ FORBIDDEN                       │
├───────────────────────────────────────┼─────────────────────────────────────┤
│  enum UserStatusEnum { ... }          │  string status = 1; // PENDING...   │
│  UserStatusEnum status = 1;           │  string type = 2; // "ADMIN","USER"│
│  Define in infra/{module}_constant    │  Enum values in comments            │
│  Shared by api/ and domain/           │  Different types across layers      │
│  TENANT_STATUS_ACTIVE                │  TENANT_STATUS_ENUM_ACTIVE         │
└───────────────────────────────────────┴─────────────────────────────────────┘
```

**Core Principle**: Enums are **FIRST-CLASS types**, NOT string conventions.

**Rules**:
1. **ALWAYS** define enum in `infra/{feature}/constant.proto`
2. **NEVER** use `string` with enum values in comments
3. **ALWAYS** use enum in both `api` and `domain` layers — same type flows through all layers
4. **NEVER** add redundant "ENUM" suffix in values: `TENANT_STATUS_ACTIVE` ✅, not `TENANT_STATUS_ENUM_ACTIVE` ❌ ,Business Meaning Only
5. **ALWAYS** use complete business meaning in enum values: `ORDER_STATUS_PENDING`, not `OS_PENDING`

### 5. Full Package AND Path Reference — MANDATORY

```proto
// ❌ FORBIDDEN — Relative reference (may cause import conflicts)
import "user_constant.proto";
message CreateUserRequest {
  UserStatusEnum status = 1;  // ❌ Which UserStatusEnum? From which package?
}

// ✅ CORRECT — Full package path
import "com/example/user/infra/user_constant.proto";
message CreateUserRequest {
  com.example.user.infra.UserStatusEnum status = 1;  // ✅ Unambiguous
}

// ✅ CORRECT — With import alias (cleaner)
import "com/example/user/infra/user_constant.proto";
package com.example.order.api;
message CreateOrderRequest {
  com.example.user.infra.UserStatusEnum user_status = 1;  // ✅ Full path
  com.example.order.infra.OrderStatusEnum order_status = 2;  // ✅ Different packages
}
```

## 📐 Layer Architecture

### Proto Package Structure — Module-First

Every feature should have a sub dir, NEVER EVER mess them together with others!

```
{package}/
├── api/
│   ├── {feature}/
│   │   ├── {feature}_service.proto    # RPC definitions
│   │   └── {feature}_dto.proto        # Request/Response messages
├── domain/
│   ├── {feature}/
│   │   ├── {feature}_entity.proto     # Database entities
└── infra/
    ├── {feature}/
    │   ├── {feature}_constant.proto   # Enums
    │   └── {feature}_error.proto      # Error codes
```

**WRONG** (don't do this):
```
{package}/
├── api/           # ❌ All APIs mixed together
├── domain/        # ❌ All entities mixed together
└── infra/         # ❌ All enums mixed together
```

### Import Rules

| From Layer  | Can Import                | Cannot Import     |
|-------------|---------------------------|-------------------|
| **api/**    | `infra/`, other `api/`    | `domain/`         |
| **domain/** | `infra/`, other `domain/` | `api/`            |
| **infra/**  | Other `infra/` only       | `api/`, `domain/` |

## 🌐 API Design (Swagger Extension)

### Import

```proto
import "apihug/protobuf/swagger/annotations.proto";
import "apihug/protobuf/swagger/swagger.proto";
import "google/protobuf/empty.proto"; //Optional(if you need)
```

### Four Extension Levels

| Level       | Extension                  | Purpose                        |
|-------------|----------------------------|--------------------------------|
| **Service** | `(hope.swagger.svc)`       | Base path, service description |
| **Method**  | `(hope.swagger.operation)` | HTTP mapping, auth, pagination |
| **Message** | `(hope.swagger.schema)`    | Object metadata                |
| **Field**   | `(hope.swagger.field)`     | Validation, examples           |

### Service-Level

```proto
service UserService {
  option (hope.swagger.svc) = {
    path: "/user";
    description: "User management service";
  };
}
```

### Operation-Level (Core)

#### HTTP Method Mapping

```proto
rpc GetUser (google.protobuf.Empty) returns (UserDetail) {
  option (hope.swagger.operation) = {
    get: "/users/{id}";
    description: "Get user by ID";
    summary: "Get User";
    tags: "user";
    operation_id: "getUserById";
    parameters: {
      parameter: {
        name: "id";
        in: PATH;
        schema: { format: INTEGER; }
      }
    }
  };
}
```

| HTTP Method | Use Case              | Example                 |
|-------------|-----------------------|-------------------------|
| `get`       | Retrieve              | `get: "/users"`         |
| `post`      | Create, complex query | `post: "/users"`        |
| `put`       | Full update           | `put: "/users/{id}"`    |
| `patch`     | Partial update        | `patch: "/users/{id}"`  |
| `delete`    | Delete                | `delete: "/users/{id}"` |

#### Pagination

```proto
rpc SearchUsers (SearchUsersRequest) returns (UserSummary) {
  option (hope.swagger.operation) = {
    post: "/users/search";
    pageable: true;  // Framework injects page/size/sort
  };
}
```

#### Batch Operations

```proto
// Batch input
rpc BatchCreate (CreateUserRequest) returns (UserSummary) {
  option (hope.swagger.operation) = {
    post: "/users/batch";
    input_repeated: true;   // Request → List<CreateUserRequest>
    output_repeated: true;  // Response → List<UserSummary>
  };
}
```

#### Authorization

**Method 1: Low-Level Risk Mode**
```proto
authorization: {
  low_limit_risky_mode: ANONYMOUS  // or LOGIN / ACTIVE
}
```

**Method 2: RBAC**
```proto
authorization: {
  rbac: {
    roles: { roles: ["ROLE_ADMIN"] };
    authorities: ["USER_CREATE", "USER_DELETE"];
    combinator: AND;  // or OR
  }
}
```

| LowLimitRiskyMode | Meaning                |
|-------------------|------------------------|
| `ANONYMOUS`       | No login required      |
| `LOGIN`           | Login required         |
| `ACTIVE`          | Login + account active |

#### Parameter Definition

**Three Rules**:
1. **PATH params** (`/{id}`) → always in `parameters{}`
2. **Query params with Request Message** → define in Message fields, NOT in `parameters{}`
3. **Simple queries (≤3 params, no body)** → use `google.protobuf.Empty` + `parameters{}`

```proto
// GET by ID — PATH param only
rpc GetUser (google.protobuf.Empty) returns (UserDetail) {
  option (hope.swagger.operation) = {
    get: "/users/{userId}";
    parameters: {
      parameter: {
        name: "userId";
        in: PATH;
        schema: { format: INTEGER; }
      }
    }
  };
}

// Complex query — Request Message + PATH
rpc SearchProducts (SearchProductsRequest) returns (ProductList) {
  option (hope.swagger.operation) = {
    get: "/tenants/{tenantId}/products";
    parameters: {
      parameter: { name: "tenantId"; in: PATH; schema: { format: INTEGER; } }
    }
  };
}

message SearchProductsRequest {
  string keyword = 1 [(hope.swagger.field) = { description: "search keyword" }];
  int32 category = 2;
}
```

#### ⚡ Request Message Design — Minimalist Principle

> **CRITICAL**: Avoid proto bloat — do NOT create Request/Response messages with only 1-2 fields unnecessarily.

**Decision Tree**:

├─────────────────────────────────────────────────────────────────────────────┤

**Examples**:

```proto
// ✅ CORRECT: 0 params — use Empty
rpc ListUsers (google.protobuf.Empty) returns (UserSummary) {
  option (hope.swagger.operation) = {
    get: "/users";
  };
}

// ✅ CORRECT: 1 param — use Empty + parameters{}
rpc SearchUsers (google.protobuf.Empty) returns (UserSummary) {
  option (hope.swagger.operation) = {
    get: "/users/search";
    parameters: {
      parameter: {
        name: "keyword";
        in: QUERY;
        schema: { type: STRING; }
      }
    };
  };
}

// ✅ CORRECT: 2 params — use Empty + parameters{}
rpc GetUserByTenant (google.protobuf.Empty) returns (UserDetail) {
  option (hope.swagger.operation) = {
    get: "/users/by-tenant";
    parameters: {
      parameter: {
        name: "tenantId";
        in: QUERY;
        schema: { format: INTEGER; }
      }
      parameter: {
        name: "userId";
        in: QUERY;
        schema: { format: INTEGER; }
      }
    };
  };
}

// ✅ CORRECT: 3+ params OR complex validation — create Request Message
rpc CreateUser (CreateUserRequest) returns (UserDetail) {
  option (hope.swagger.operation) = {
    post: "/users";
  };
}

message CreateUserRequest {
  string username = 1 [(hope.swagger.field) = {
    description: "User login name";
    empty: false;
    min_length: 3;
    max_length: 32;
  }];
  string email = 2 [(hope.swagger.field) = {
    description: "User email";
    empty: false;
    email: true;
  }];
  int32 age = 3 [(hope.swagger.field) = {
    description: "User age";
    minimum: 0;
    maximum: 150;
  }];
  UserStatusEnum status = 4;
}

// ❌ FORBIDDEN: Request Message with only 1 field (proto bloat)
rpc SearchUsers (SearchUsersRequest) returns (UserSummary) {
  option (hope.swagger.operation) = {
    get: "/users/search";
  };
}
message SearchUsersRequest {
  string keyword = 1;  // ❌ Only 1 field — use Empty + parameters{} instead!
}

// ❌ FORBIDDEN: Request Message with only 2 fields (proto bloat)
rpc GetUser (GetUserRequest) returns (UserDetail) {
  option (hope.swagger.operation) = {
    get: "/users/{id}";
  };
}
message GetUserRequest {
  int64 id = 1;           // ❌ Only 1 field (PATH param) — use Empty!
  string include = 2;     // ❌ Total 2 fields — still too few for Message
}
```

**Why This Matters**:

| Problem                      | Impact                                                                               |
|------------------------------|--------------------------------------------------------------------------------------|
| **Proto Bloat**              | Hundreds of single-field Request messages → hard to maintain                         |
| **Code Generation Overhead** | Unnecessary Java/JS classes generated                                                |
| **API Readability**          | `rpc GetUser (google.protobuf.Empty)` is clearer than `rpc GetUser (GetUserRequest)` |
| **Consistency**              | Framework convention — follow the crowd                                              |

**Rule of Thumb**:
> **3-Field Rule**: If your Request Message has fewer than 3 fields, reconsider using `google.protobuf.Empty` + `parameters{}`.

### Naming Conventions

| Element    | Pattern                                  | Example                                |
|------------|------------------------------------------|----------------------------------------|
| **Table**  | `{PREFIX}_{ENTITY}`                      | `SYS_USER`, `BIZ_ORDER`, `CFG_SETTING` |
| **Column** | `{ENTITY}_{FIELD}` or `{FIELD}_{SUFFIX}` | `USER_NAME`, `ORDER_NO`, `STATUS_CODE` |
| **Index**  | `IDX_{TABLE}_{COLUMN}`                   | `IDX_USER_EMAIL`                       |
| **Unique** | `UK_{TABLE}_{COLUMN}`                    | `UK_USER_EMAIL`                        |

**Table Prefixes**:
| Prefix | Domain | Example |
|--------|--------|---------|
| `SYS_` | System/Platform | `SYS_USER`, `SYS_ROLE` |
| `BIZ_` | Business | `BIZ_ORDER`, `BIZ_PRODUCT` |
| `CFG_` | Configuration | `CFG_SETTING` |
| `LOG_` | Logging/Audit | `LOG_OPERATION` |
| `TMP_` | Temporary | `TMP_IMPORT` |

### Table Extension

```proto
message UserEntity {
  option (hope.persistence.table) = {
    name: "SYS_USER";
    description: "User information table";
    wires: [IDENTIFIABLE, AUDITABLE, DELETABLE, TENANTABLE];

    unique_constraints: {
      name: "UK_USER_EMAIL";

    indexes: {
      name: "IDX_USER_TENANT";
      column_list: ["tenant_id"];

    // Liquibase customization (rarely needed)
    liquibase: {
      version: 1;
      comment: "Fulltext index for MySQL";

      sql: ["CREATE FULLTEXT INDEX IDX_USER_NAME_FT ON SYS_USER(USER_NAME)"];
    }
  };

  // Fields...
}
```

#### Wires (Built-in Features)

| Wire           | Auto-Added Fields                                      |
|----------------|--------------------------------------------------------|
| `IDENTIFIABLE` | `id` (Long)                                            |
| `AUDITABLE`    | `created_at`, `updated_at`, `created_by`, `updated_by` |
| `DELETABLE`    | `deleted` (Boolean, soft-delete)                       |
| `TENANTABLE`   | `tenant_id` (Long, multi-tenancy)                      |
| `VERSIONABLE`  | `version` (Integer, optimistic locking)                |
| `ALL`          | All above combined                                     |

```proto
wires: [IDENTIFIABLE, AUDITABLE, DELETABLE]  // Common combination
```

## 🔄 DB Migration (Liquibase - Auto-Generated)

> **CRITICAL**: Liquibase migrations are 100% auto-generated from proto entity definitions. NEVER create or modify migration files manually.

### Location Structure

```
src/main/resources/liquibase/{domain}/
├── master.xml              ← Register all versions
└── {version}/
    ├── _before.xml         ← Pre-migration hooks (handwritten)
    ├── _full.xml           ← ⚠️ AUTO-GENERATED — NEVER EDIT
    ├── _partial.xml        ← Incremental changes (handwritten)
    └── _after.xml          ← Post-migration corrections (handwritten)
```

### Absolute Rules (ZERO TOLERANCE)

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  LIQUIBASE MIGRATION — IMMUTABLE RULES                                          │
├─────────────────────────────────────────────────────────────────────────────────┤
│  ❌ NEVER modify _full.xml (auto-generated from proto)                         │
│  ❌ NEVER modify _before.xml / _partial.xml (handwritten hooks)                │
│  ✅ Corrections → add changeSet to _after.xml with <preConditions>             │
│  ✅ New version → create directory + register in master.xml                    │
│  ✅ Trigger: gradlew :{module}:wire (auto-generates from proto)                │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### File Responsibility

| File           | Type       | Purpose                                 | Editable                         |
|----------------|------------|-----------------------------------------|----------------------------------|
| `master.xml`   | Registry   | Register all migration versions         | ✅ Manual (register new versions) |
| `_full.xml`    | Schema     | Complete database schema from proto     | ❌ **AUTO-GENERATED**             |
| `_before.xml`  | Hook       | Pre-migration operations                | ✅ Handwritten (rare)             |
| `_partial.xml` | Delta      | Incremental changes                     | ✅ Handwritten (rare)             |
| `_after.xml`   | Correction | Post-migration fixes with preconditions | ✅ **Main place for corrections** |

### Correction Pattern

**When you need to fix or add fields not covered by proto:**

```xml
<!-- ✅ CORRECT: Add correction to _after.xml with preConditions -->
<databaseChangeLog xmlns="http://www.liquibase.org/xml/ns/dbchangelog"
                   xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
                   xsi:schemaLocation="http://www.liquibase.org/xml/ns/dbchangelog
                                       http://www.liquibase.org/xml/ns/dbchangelog/dbchangelog-4.20.xsd">

  <changeSet id="20260311-001" author="developer">
    <preConditions onFail="MARK_RAN">
      <tableExists tableName="SYS_USER"/>
      <not><columnExists tableName="SYS_USER" columnName="extra_field"/></not>
    </preConditions>
    <addColumn tableName="SYS_USER">
      <column name="extra_field" type="VARCHAR(64)" defaultValue="DEFAULT_VALUE">
        <constraints nullable="false"/>
      </column>
    </addColumn>
  </changeSet>

</databaseChangeLog>
```

**Why `<preConditions onFail="MARK_RAN">`:**
- Ensures idempotency — migration won't fail if change already applied
- Marks changeSet as "run" even if precondition fails (prevents re-execution)

### Workflow

```
1. Modify proto entity (add/rename/change field)
   ↓
2. Run: gradlew :{module}:wire
   ↓
3. wire auto-generates:
   - Entity classes (src/generated/domain/)
   - _full.xml (complete schema in liquibase/{domain}/{version}/)
   - Updates master.xml reference
   ↓
4. If correction needed:
   - Add changeSet to _after.xml with <preConditions>
   - DO NOT modify _full.xml
```

### ⚠️ Common Mistakes

```xml
<!-- ❌ FORBIDDEN: Modify _full.xml directly -->
<changeSet id="full" author="wire">
  <!-- Manual edits here will be OVERWRITTEN by next wire run -->
</changeSet>

<!-- ❌ FORBIDDEN: Add changeSet without preConditions -->
<changeSet id="add-column" author="developer">
  <addColumn tableName="SYS_USER">  <!-- Fails if column already exists! -->
    <column name="extra_field" type="VARCHAR(64)"/>
  </addColumn>
</changeSet>

<!-- ✅ CORRECT: Use preConditions for idempotency -->
<changeSet id="add-column" author="developer">
  <preConditions onFail="MARK_RAN">
    <tableExists tableName="SYS_USER"/>
    <not><columnExists tableName="SYS_USER" columnName="extra_field"/></not>
  </preConditions>
  <addColumn tableName="SYS_USER">
    <column name="extra_field" type="VARCHAR(64)"/>
  </addColumn>
</changeSet>
```

### Version Management

**Creating a new migration version:**

```
src/main/resources/liquibase/{domain}/
├── master.xml              ← Add new version reference
└── 0.0.1-RELEASE/               ← Version directory (YYYYMMDD format)
    ├── _before.xml
    ├── _full.xml           ← Generated by wire
    ├── _partial.xml

```

**master.xml registration:**
```xml

```

## 📋 Quick Reference

### Decision Tables

#### API Design

| Scenario        | Proto Setting           | Request           | Response         |
|-----------------|-------------------------|-------------------|------------------|
| Paginated query | `pageable: true`        | Query fields only | Single item type |
| Simple list     | `output_repeated: true` | Normal            | Single item type |
| Batch input     | `input_repeated: true`  | Single item type  | Normal           |
| Batch I/O       | Both                    | Single item type  | Single item type |

#### Database Modeling

| Requirement      | Wire Selection                      |
|------------------|-------------------------------------|
| Primary key only | `wires: [IDENTIFIABLE]`             |
| + Audit fields   | `wires: [IDENTIFIABLE, AUDITABLE]`  |
| + Soft delete    | `wires: [IDENTIFIABLE, DELETABLE]`  |
| + Multi-tenancy  | `wires: [IDENTIFIABLE, TENANTABLE]` |
| All features     | `wires: [ALL]`                      |

#### Type Selection

| Data Type          | Proto → SQL                                |
|--------------------|--------------------------------------------|
| Short string       | `string` → `VARCHAR(length)`               |
| Long text          | `string` → `CLOB`                          |
| Integer            | `int32` → `INTEGER`                        |
| Large integer / ID | `int64` → `BIGINT`                         |
| Decimal / Money    | `double` → `DECIMAL(precision, scale)`     |
| Boolean            | `bool` → `BOOLEAN`                         |
| Date/Time          | `string` → `DATE` / `TIMESTAMP`            |
| Enum               | `EnumType` → `VARCHAR + enum_type: STRING` |
